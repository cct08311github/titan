# TITAN — 上線後監控計畫（Post-Launch Plan）

> T26 | 對應 Issue #37

---

## 1. 概覽

本計畫涵蓋 TITAN 平台正式上線後兩週的密集監控期，包含：
- 每日監控任務
- 事件回應程序
- 使用者回饋收集
- 第二週穩定化評估
- 經驗教訓模板

**監控期間：** Go-Live 後第 1–14 天
**負責人：** DevOps Lead + QA Lead

---

## 2. 兩週監控計畫

### 第一週（密集監控期，Day 1–7）

**監控頻率：** 每 4 小時一次

#### Day 1（上線當天）

| 時段 | 任務 | 負責人 |
|------|------|--------|
| 08:00 | 執行完整煙霧測試 | DevOps |
| 09:00 | 確認第一批使用者登入成功 | QA |
| 12:00 | 查看 Grafana Dashboard，確認指標正常 | DevOps |

| 17:00 | Day 1 每日報告 | DevOps + PM |
| 持續 | 監控 Grafana 告警（每 4 小時） | DevOps |

#### Day 2–5（工作日密集期）

每日必做任務：

- [ ] 08:00 — 煙霧測試（`./scripts/sit-smoke-test.sh`）

- [ ] 17:00 — 填寫每日狀態報告（見 2.3 節）

#### Day 6–7（週末）

- [ ] 每 8 小時執行一次煙霧測試
- [ ] 確認週末備份正常
- [ ] 值班人員隨時待命（電話聯繫）

## 3. 事件回應程序

### 3.1 事件分級

| 等級 | 定義 | 初始回應時間 | 解決時限 |
|------|------|-------------|----------|
| P0（緊急） | 服務完全不可用、資料遺失 | **15 分鐘** | **4 小時** |
| P1（高） | 核心功能異常、大量使用者受影響 | **30 分鐘** | **8 小時** |
| P2（中） | 部分功能異常、少數使用者受影響 | **2 小時** | **24 小時** |
| P3（低） | 輕微問題、使用者體驗下降 | **下一工作日** | **3 工作日** |

### 3.2 事件回應流程

```
監控告警觸發
    │
    ▼
值班人員確認（15 分鐘內）
    │
    ├── P0/P1 → 立即通知技術主管 + PM
    │               │
    │               ▼
    │           建立 Incident Channel（通訊群組）
    │               │
    │               ▼
    │           指定 Incident Commander
    │               │
    │               ▼
    │           診斷根因（Root Cause Analysis）
    │               │
    │               ├── 可快速修復 → 實施修復 → 驗證 → 關閉事件
    │               │
    │               └── 需回滾 → 執行 cutover-runbook.md 回滾步驟
    │
    └── P2/P3 → 建立 GitHub Issue → 排入修復計畫
```

## 5. 第 14 天評估與穩定化報告

### 5.1 評估標準

若以下所有條件成立，進入常規維運模式（降低監控密度）：

| 條件 | 標準 | 結果 |
|------|------|------|
| 服務可用性 | ≥ 99.5%（兩週平均） | [ ] 達標 / [ ] 未達標 |
| P0 事件 | 0 個 | [ ] 達標 / [ ] 未達標 |
| P1 事件 | ≤ 2 個（均已解決） | [ ] 達標 / [ ] 未達標 |
| 使用者滿意度 | 平均分 ≥ 4.0/5.0 | [ ] 達標 / [ ] 未達標 |
| 開放工單 | ≤ 5 個（均為 P2/P3） | [ ] 達標 / [ ] 未達標 |
| 備份成功率 | 100% | [ ] 達標 / [ ] 未達標 |

若任一條件未達標，延長密集監控期一週，並制定改善計畫。
