# TITAN SLA 服務水準協議

**文件版本**: v1.0
**撰寫日期**: 2026-03-24
**適用對象**: 資訊長（CIO）、資訊部門主管、L1/L2 支援團隊
**審核週期**: 每半年複審一次

---

## 1. 概述

本文件定義 TITAN 工作管理系統的服務水準協議（Service Level Agreement, SLA），涵蓋系統可用性、回應時間、事件處理流程及維護窗口。本 SLA 適用於銀行資訊部門內部 IT 團隊（共 5 人），部署環境為封閉內網。

---

## 2. 系統可用性目標

### 2.1 可用性等級

TITAN 為內部工作管理工具，非關鍵業務系統（非交易核心、非客戶面對系統），依此定義可用性目標：

| 指標 | 目標值 | 計算基礎 |
|------|--------|---------|
| 月可用性 | ≥ 99.5% | 扣除計劃維護時間後計算 |
| 年可用性 | ≥ 99.2% | 允許年計劃外停機 ≤ 63 小時 |
| 計劃外停機上限 | ≤ 3.6 小時/月 | |
| 計劃維護時間 | 不計入停機 | 須提前 24 小時公告 |

### 2.2 可用性計算公式

```
可用性 (%) = (總服務時間 - 計劃外停機時間) / 總服務時間 × 100

其中：
- 總服務時間 = 當月日曆時間（小時）- 計劃維護時間
- 計劃外停機時間 = P1 + P2 事件造成服務完全不可用的時間
```

### 2.3 服務時間定義

- **核心服務時間**：週一至週五 08:00-19:00（正常上班時段）
- **延伸服務時間**：週一至週五 19:00-22:00（允許較長回應時間）
- **非服務時間**：週六、週日及國定假日（停機不計入 SLA）

> 注意：TITAN 為 7x24 技術上可用（系統持續運行），但 SLA 監控以核心服務時間為主。

## 4. 事件分級與回應時間

### 4.1 事件嚴重等級定義

**P1 — 緊急（Critical）**

定義：系統完全無法使用，影響所有使用者，或發生資安事件。

範例：
- TITAN 主服務（Next.js）無法啟動
- 資料庫（PostgreSQL）連線失敗
- 偵測到未授權存取或資料外洩
- 所有使用者無法登入

回應要求：
- **初始回應**：15 分鐘（核心服務時間）
- **問題確認**：30 分鐘
- **復原目標（RTO）**：2 小時
- 通知對象：IT 主管、資安主管

---

**P2 — 嚴重（High）**

定義：主要功能部分不可用，影響多數使用者正常作業。

範例：
- 任務建立或更新功能失效
- 報表或搜尋功能完全無回應
- 系統回應時間超過 SLA 5 倍以上
- Email/通知功能持續失敗

回應要求：
- **初始回應**：1 小時（核心服務時間）
- **問題確認**：2 小時
- **復原目標（RTO）**：4 小時
- 通知對象：IT 主管

---

**P3 — 一般（Medium）**

定義：部分功能異常，影響少數使用者，有替代方案可繞行。

範例：
- 特定頁面顯示錯誤但其他功能正常
- 效能降低但仍在可用範圍
- 部分附件上傳失敗
- UI 顯示異常（版面跑版等）

回應要求：
- **初始回應**：4 小時（核心服務時間）
- **修復目標**：下一個工作日
- 通知對象：當班工程師

---

**P4 — 低優先（Low）**

定義：輕微問題，不影響核心功能，屬改善性需求。

範例：
- 文字錯誤或翻譯問題
- 非關鍵 UI 優化需求
- 效能微調（在 SLA 範圍內）

回應要求：
- **初始回應**：下一個排程 Sprint
- **修復目標**：視 backlog 優先度排程

---

### 4.2 事件回應時間摘要

| 等級 | 初始回應 | 問題確認 | 復原/修復目標 | 通知對象 |
|------|---------|---------|-------------|---------|
| P1 | 15 分鐘 | 30 分鐘 | 2 小時 | IT 主管、資安主管 |
| P2 | 1 小時 | 2 小時 | 4 小時 | IT 主管 |
| P3 | 4 小時 | 8 小時 | 次日 | 當班工程師 |
| P4 | N/A | N/A | Sprint 排程 | 無 |

## 5. 維護窗口定義

### 5.1 計劃維護時間

| 類型 | 時間 | 頻率 | 提前公告 |
|------|------|------|---------|
| 例行維護 | 週三 22:00-24:00 | 每月 1 次 | 72 小時前 |
| 安全更新 | 週五 22:00-01:00 | 依需求（≤每月 1 次）| 48 小時前 |
| 重大版本升級 | 週六 08:00-17:00 | 依專案計畫 | 1 週前 |
| 緊急修補 | 任何時間 | 依需求 | 30 分鐘前（事前通知）|

### 5.2 維護公告方式

維護作業執行前，須透過以下方式通知全體使用者：
1. 在 TITAN 系統首頁顯示維護公告橫幅
2. 發送 Email 至全體 IT 部門成員
3. 於 IT 部門 Discord/即時通訊頻道發佈通知

### 5.3 緊急維護處理

若發生需要立即執行的緊急維護（如 P1 資安事件），流程如下：
1. IT 主管授權後立即執行
2. 同步發送緊急通知（Email + 即時通訊）
3. 維護期間每 30 分鐘更新狀態
4. 維護完成後 1 小時內發送事件報告

## 6. 升級流程

### 6.1 事件升級觸發條件

| 觸發條件 | 升級動作 |
|---------|---------|
| P2 事件超過 2 小時未解決 | 升級為 P1，通知 IT 主管 |
| P3 事件超過 1 個工作日未回應 | 升級為 P2 |
| 同類問題在 1 個月內發生 3 次以上 | 要求 RCA（Root Cause Analysis）|
| 使用者反映影響範圍擴大 | 重新評估事件等級 |
| 發現資安相關跡象 | 立即升級 P1，通知資安主管 |

### 6.2 升級通報鏈

```
一線工程師
    ↓（30分鐘未解決）
資深工程師 / 值班主管
    ↓（1小時未解決）
IT 主管
    ↓（P1 或資安事件）
資安主管 / CIO
```

### 6.3 事件後續作業

- **P1/P2 事件**：事件解決後 24 小時內完成 RCA 初稿，72 小時內完成正式報告
- **P3 事件**：記錄於 TITAN 的問題追蹤系統，納入下一 Sprint 處理
- **重複性事件**：3 次以上需提出系統性改善方案

## 7. SLA 監控與報告

### 7.1 監控機制

- **即時監控**：使用 Grafana + Prometheus 監控系統健康指標（CPU、記憶體、回應時間）
- **可用性計算**：由 Prometheus 自動記錄 uptime，每月月底自動產生報告
- **效能告警**：API 回應時間超過 P95 目標時自動發送告警

### 7.2 定期報告

| 報告類型 | 頻率 | 內容 | 接收對象 |
|---------|------|------|---------|
| 月度 SLA 報告 | 每月 5 日前 | 可用性、事件統計、回應時間達成率 | IT 主管、CIO |
| 季度效能回顧 | 每季末 | SLA 達成趨勢、改善項目 | CIO |
| 年度 SLA 複審 | 每年 1 月 | SLA 目標適當性評估、更新建議 | CIO、相關主管 |

---

## 8. SLA 適用限制

以下情況不計入 SLA 統計：

1. **外部因素**：網路骨幹設備故障、機房斷電（非 TITAN 系統責任範圍）
2. **使用者端問題**：個人電腦故障、瀏覽器相容性問題
3. **計劃維護**：已公告的維護窗口
4. **不可抗力**：颱風、地震等天災導致的服務中斷
5. **第三方服務**：依賴的外部 API 或服務中斷（若有）

---

## 9. SLA 違約處理

由於 TITAN 為內部系統，SLA 違約不適用財務懲罰機制，但：

- P1 SLA 未達成 → 須提交事件報告至 IT 主管，並於下次管理會議說明改善措施
- 月可用性低於 99.0% → 須提交月度改善計畫
- 連續 2 個月可用性低於目標 → 觸發系統健康評估，考慮架構調整
