# Part 1：Top 10 產品 UX 研究

## 1.1 研究框架

針對每個產品，分析四個維度：
- **輸入方式**：使用者如何記錄時間？
- **操作步驟**：最少幾次互動完成一筆記錄？
- **視圖呈現**：日/週/月檢視長什麼樣？
- **Aha Moment**：什麼設計讓人覺得「好用」？

---

## 1.2 逐產品分析

### 1.2.1 Toggl Track — 極簡計時器之王

| 維度 | 分析 |
|------|------|
| **輸入方式** | Timer 模式：一鍵開始/停止，自動記錄起訖時間。手動模式：輸入起訖時間（09:00-12:00）或直接填時長（3h）。支援兩種模式切換。 |
| **操作步驟** | Timer：2 clicks（開始+停止）。手動：3 步（描述→起訖時間→專案）。 |
| **視圖呈現** | **日檢視**：時間列表，每筆顯示起訖時間+時長+專案色條。**週檢視**：橫向 timeline，每天一列，色塊代表不同專案。**月報**：圓餅圖+長條圖，按專案/客戶分組。 |
| **Aha Moment** | 「起訖時間可直接在列表上點擊修改」— 不需要開彈窗，直接 inline 改時間。週 timeline 的色塊排列直覺顯示時間分布。 |

**對 TITAN 的啟發**：起訖時間的 inline 編輯模式值得借鏡。Timer 停止後自動填入起訖時間的體驗非常流暢。

### 1.2.2 Clockify — 免費之王，格子最佳

| 維度 | 分析 |
|------|------|
| **輸入方式** | 雙模式：Timer bar（頂部常駐）+ Timesheet grid（週格子直接輸入時長）。手動可填起訖時間或僅填時長。 |
| **操作步驟** | Timesheet 格子：1 click + type = 最少步驟。Timer：2 clicks。新增行：選專案→選任務→填格子 = 3 步。 |
| **視圖呈現** | **Timesheet 模式**：週一~日格子，每列=專案+任務，格子內填數字。**Timer 模式**：時間列表，每筆有起訖時間。**Calendar 檢視**（付費）：類似 Google Calendar 的日/週視圖，時間區塊可拖拉。**Dashboard**：圓餅圖+趨勢線。 |
| **Aha Moment** | 「格子裡直接 type 數字，Tab 跳下一格」— 像 Excel 一樣快。付費版的 Calendar 視圖則是另一個 aha：拖拉建立時間區塊。 |

**對 TITAN 的啟發**：Clockify 證明了「格子模式」和「日曆模式」可以共存。v2 的格子設計仍然有價值，v3 應該是「新增日曆視圖」而非「取代格子」。

### 1.2.3 Harvest — 乾淨設計，專案導向

| 維度 | 分析 |
|------|------|
| **輸入方式** | 以「日」為主要單位。每天一頁，列出專案/任務列表，每列有 Timer 按鈕和時長輸入框。**不支援起訖時間**，只有時長。 |
| **操作步驟** | 已有列：1 click + type。新增：選專案→選任務→輸入時長 = 3 clicks。 |
| **視圖呈現** | **日檢視**：主畫面，每個專案一列，右邊顯示時長+Timer。**週檢視**：格子表格，結構同 Clockify。**無月曆視圖**，但有月報圖表。 |
| **Aha Moment** | 「一天一頁、大字體、高對比色」— 視覺清爽不擁擠。審批流程（Submit → Approve）整合得很自然。 |

**對 TITAN 的啟發**：Harvest 證明「日檢視」是手機上最好的工時輸入介面。其審批流程整合方式值得參考。

### 1.2.4 Jira Tempo — 企業級，與任務深度整合

| 維度 | 分析 |
|------|------|
| **輸入方式** | 三種入口：(1) 在 Jira Issue 上直接 Log Work（起訖時間或時長）。(2) Tempo Timesheet 格子（週格子，直接填數字）。(3) Tempo Planner（日曆拖拉規劃工時）。 |
| **操作步驟** | 格子：1 click + type。Issue Log Work：開 dialog → 填時長 → 儲存 = 3 clicks。Planner：拖拉 = 2 動作。 |
| **視圖呈現** | **Timesheet**：週格子+日合計+週合計。**日曆視圖（Planner）**：像 Google Calendar，可拖拉任務到時段。**報表**：按團隊/專案/成員多維分析。 |
| **Aha Moment** | 「從 Jira Issue 直接記工時，不用切換畫面」— 對已經用 Jira 的團隊是殺手功能。Planner 的日曆拖拉也很直覺。 |

**對 TITAN 的啟發**：Tempo Planner 的日曆拖拉設計是 TITAN v3 的重要參考。「從任務頁直接記工時」的概念也值得引入。

### 1.2.5 Monday.com — 視覺化工作管理

| 維度 | 分析 |
|------|------|
| **輸入方式** | 在任務列上新增 Time Tracking column，直接啟動 Timer 或手動輸入時長。支援起訖時間設定。時間記錄附著在任務行上。 |
| **操作步驟** | Timer：2 clicks（在任務行上點開始/停止）。手動：點開 time column → 輸入 = 2 clicks。 |
| **視圖呈現** | **Table View**：每列一個任務，Time column 顯示累計時長。**Timeline View**：甘特圖式，非工時用。**Workload View**：堆疊長條圖顯示每人每週工時分配。**無獨立日曆工時視圖**。 |
| **Aha Moment** | 「時間記錄就在任務旁邊，不用跳到另一個模組」— 降低認知負載。色彩和圖示用得極好，視覺辨識度高。 |

**對 TITAN 的啟發**：「時間記錄附著在任務上」的概念強化了 TITAN 的任務-工時關聯。色彩系統設計值得學習。

### 1.2.6 ClickUp Time Tracking — All-in-One

| 維度 | 分析 |
|------|------|
| **輸入方式** | Global Timer（頂部常駐，任何頁面可啟動）。任務內 Time Tracking 面板。手動輸入支援起訖時間或時長。Timesheet 檢視（格子）。 |
| **操作步驟** | Global Timer：1 click 開始。手動：3 clicks（進入任務→開 panel→填入）。Timesheet 格子：1 click + type。 |
| **視圖呈現** | **Timesheet**：週格子，類似 Clockify。**Timeline**：甘特圖。**日曆視圖**：任務排程用，非工時專用。**Workload**：團隊工時分配。 |
| **Aha Moment** | 「Global Timer 不會隨頁面切換消失」— 非常實用。Timesheet 可在格子和列表間切換。 |

**對 TITAN 的啟發**：Global Timer（sticky bar）的設計 v2 已採納。ClickUp 的格子/列表切換概念可擴展為格子/列表/日曆三模式切換。

### 1.2.7 Timely — 自動追蹤先驅

| 維度 | 分析 |
|------|------|
| **輸入方式** | **自動追蹤**：背景記錄使用者活動（app usage、URL、文件），AI 生成時間建議。使用者只需「確認」或「調整」。手動模式：在日曆上拖拉建立時間區塊。 |
| **操作步驟** | 自動模式：1 click（確認 AI 建議）。手動：拖拉建立 + 選專案 = 2 步。 |
| **視圖呈現** | **核心視圖就是日曆**：日/週視圖，時間軸上的色塊代表不同專案。像 Google Calendar 但專為工時設計。**Timeline**：橫向時間流。**Memory Timeline**：顯示 AI 偵測到的活動，拖拉到日曆上確認。 |
| **Aha Moment** | 「日曆就是工時表」— 不需要另一個 timesheet 頁面，日曆本身就是記錄工時的地方。AI 建議大幅降低手動輸入。 |

**對 TITAN 的啟發**：Timely 是「日曆即工時表」概念的最佳實踐。其日/週日曆視圖的拖拉互動是 TITAN v3 的核心參考。

### 1.2.8 RescueTime — 被動追蹤

| 維度 | 分析 |
|------|------|
| **輸入方式** | **完全被動**：背景 agent 自動記錄所有 app/網站使用時間。使用者不需要任何輸入動作。可手動標記「Focus Time」區段。 |
| **操作步驟** | 0 clicks（自動）。手動標記：2 clicks。 |
| **視圖呈現** | **Dashboard**：當日/本週概覽，堆疊長條圖+生產力分數。**日檢視**：時間軸，色塊代表 productive/neutral/distracting。**週/月**：趨勢圖，每日工時長條圖。**無格子/日曆編輯**。 |
| **Aha Moment** | 「完全不用動手，打開就看到結果」— 極致的低摩擦。生產力分數提供即時回饋。 |

**對 TITAN 的啟發**：RescueTime 的「生產力分數」概念可轉化為 TITAN 的「任務投入率」指標。其趨勢圖設計在月報中值得借鏡。

### 1.2.9 Google Calendar — 日曆 UX 的黃金標準

| 維度 | 分析 |
|------|------|
| **輸入方式** | **拖拉建立**：在時間軸上按住拖拉，自動建立事件（例：9:00 拖到 12:00 = 3 小時區塊）。**點擊快速新增**：點擊空白區域，彈出精簡建立框。**鍵盤快捷鍵**：按 C 快速建立。 |
| **操作步驟** | 拖拉：1 動作（拖）+ 輸入標題 + Enter = 3 步。點擊：1 click + 輸入標題 + 設定時間 = 3 步。 |
| **視圖呈現** | **日檢視**：24 小時垂直時間軸，每 15 分鐘一格。事件以色塊呈現。**週檢視**：7 欄日曆，每欄同日檢視。頂部顯示全天事件。**月檢視**：格子日曆，每天顯示事件標題列表。點擊日期展開日檢視。**排程檢視**：多人水平時間軸。 |
| **Aha Moment** | 「拖拉建立」是革命性的。看到時間被「佔據」的視覺化效果，比數字更直覺。色塊大小直接反映時間長度。resize handle 可拖拉調整時長。 |

**對 TITAN 的啟發**：Google Calendar 的拖拉建立、色塊呈現、resize handle 是 TITAN v3 日曆視圖的核心設計基礎。月檢視的「點擊展開」交互也是最佳範例。

### 1.2.10 Apple Calendar — 原生手勢互動

| 維度 | 分析 |
|------|------|
| **輸入方式** | 與 Google Calendar 類似：拖拉建立、點擊新增。差異在於原生手勢：long press 啟動建立、3D Touch peek 預覽。時間選擇器使用 iOS 原生 scroll wheel。 |
| **操作步驟** | 拖拉：1 動作。Long press + 填入：3 步。 |
| **視圖呈現** | **日檢視**：垂直時間軸，事件色塊圓角設計。**週檢視**：5 或 7 欄（可設定）。**月檢視**：上半螢幕日曆格子 + 下半螢幕當日事件列表。**年檢視**：12 月小日曆+熱力圖。 |
| **Aha Moment** | 「月檢視的上下分割」— 上面選日期、下面看詳情，一個畫面完成兩件事。拖拉事件在日期間移動（reschedule）非常流暢。 |

**對 TITAN 的啟發**：Apple Calendar 的月檢視分割設計適合 TITAN 的月報：上方日曆格子（顯示每日總時數+色彩指標）+ 下方當日詳情。手機版的 long press 建立手勢也值得採用。

## 1.3 綜合比較矩陣

| 產品 | 起訖時間 | 時長輸入 | 日曆拖拉 | 格子模式 | 日檢視 | 週檢視 | 月檢視 | 最少步驟 |
|------|:---:|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| Toggl Track | 有 | 有 | 無 | 無 | 列表 | Timeline | 圖表 | 2 |
| Clockify | 有 | 有 | 有（付費）| 有（最佳）| 列表 | 格子/日曆 | 圖表 | 1+type |
| Harvest | 無 | 有 | 無 | 有 | 列表 | 格子 | 圖表 | 1+type |
| Jira Tempo | 有 | 有 | 有（Planner）| 有 | 日曆 | 格子/日曆 | 報表 | 1+type |
| Monday.com | 有 | 有 | 無 | 表格 | 無 | Workload | 無 | 2 |
| ClickUp | 有 | 有 | 無 | 有 | 無 | 格子 | 無 | 1+type |
| Timely | 有 | 有 | 有（核心）| 無 | 日曆 | 日曆 | Timeline | 1（AI）|
| RescueTime | 自動 | 自動 | 無 | 無 | 時間軸 | 圖表 | 圖表 | 0 |
| Google Calendar | 有 | N/A | 有（核心）| N/A | 日曆 | 日曆 | 格子 | 1 拖拉 |
| Apple Calendar | 有 | N/A | 有（核心）| N/A | 日曆 | 日曆 | 分割 | 1 拖拉 |

### 關鍵發現

1. **起訖時間 vs 時長**：專業工時工具（Toggl、Clockify、Tempo）都支援兩種模式並存，讓使用者自由選擇。
2. **日曆拖拉**：三個產品以日曆為核心（Timely、Google Calendar、Apple Calendar），兩個產品作為進階功能（Clockify 付費、Tempo Planner）。
3. **格子模式仍有價值**：Clockify、Harvest、Tempo、ClickUp 的週格子都是高效的批次輸入方式，不應完全捨棄。
4. **最佳策略 = 多模式切換**：Clockify 和 Tempo 都提供「格子 ↔ 日曆」切換，滿足不同場景需求。

# Part 2：TITAN v3 日曆視圖設計

## 2.1 設計原則

> **「你的工時表，看起來像你的行事曆。」**

### 核心理念

1. **時間是視覺化的**：一個 3 小時的區塊就是比 1 小時的區塊大 3 倍，不需要讀數字就能感受時間分配。
2. **起訖時間是自然的**：人們思考的是「我 9 點到 12 點在做 A 專案」，而非「我做了 3 小時 A 專案」。
3. **多模式共存**：日曆視圖是新增的選項，格子視圖保留給習慣批次填寫的使用者。

### 視圖架構

### 互動設計

#### 拖拉建立（Drag to Create）

```
操作流程：
1. 在時間軸空白處按下滑鼠（例：09:00 位置）
2. 按住往下拖拉到目標時間（例：12:00 位置）
3. 放開滑鼠 → 出現預建立面板

預建立面板（Popover，出現在區塊旁邊）：
+-------------------------------+
| 新增工時紀錄                    |
| 時間：09:00 — 12:00 (3h)      |
| 任務：[選擇任務...        v]   |
| 分類：[原始規劃           v]   |
| 備註：[___________________]    |
|          [取消]  [儲存]        |
+-------------------------------+

快速儲存：如果不需要改任何東西，
選好任務後按 Enter 即可 = 3 步完成
```

#### 點擊快速新增（Quick Add）

```
操作流程：
1. 點擊時間軸空白處（例：14:00）
2. 出現快速新增面板

+-------------------------------+
| + 新增                         |
| 開始：[14:00]  結束：[15:00]   |
| 任務：[選擇任務...        v]   |
|          [更多選項]  [儲存]    |
+-------------------------------+

預設 1 小時，可調整。「更多選項」展開完整編輯。
```

#### 區塊互動

```
點擊區塊 → 展開詳情卡片：
+-----------------------------------+
| # TITAN-123 前端重構               |
| 09:00 — 12:00  |  3h              |
| ● 原始規劃                         |
| 備註：完成 Component A 重構        |
|                                    |
| [編輯] [複製] [刪除]   審核：待審  |
+-----------------------------------+

拖拉區塊上下邊緣 → 調整起訖時間（resize）

```

#### 從左側邊欄拖拉任務

```
操作流程：
1. 從左側「待填任務」清單拖拉一個任務卡片
2. 拖到右邊時間軸的目標時段
3. 放下 → 自動建立 1 小時區塊（可 resize 調整）
```

## 2.7 資料模型變更

### 現有 Schema（已支援）

```prisma
model TimeEntry {
  // ... 現有欄位
  startTime DateTime?   // ← 已存在，目前僅 Timer 使用
  endTime   DateTime?   // ← 已存在，目前僅 Timer 使用
  hours     Float       // ← 已存在，v3 由起訖時間自動計算
}
```

### v3 的使用方式

| 情境 | startTime | endTime | hours | 說明 |
|------|-----------|---------|-------|------|
| 日曆拖拉建立 | 有 | 有 | 自動算 | 起訖時間 → hours |
| 快速新增（含起訖時間）| 有 | 有 | 自動算 | 同上 |
| Timer 停止 | 有 | 有 | 自動算 | 同上 |
| 格子填入（僅時數）| null | null | 手動輸入 | 向下相容 v2 |
| Kimai 匯入 | 有 | 有 | 自動算 | Kimai 原本就有起訖時間 |

**結論：不需要 Schema 變更**，現有 `startTime`/`endTime` 已足夠。需要的是前端和 API 的邏輯調整。

### API 變更

```typescript
// 現有 createTimeEntrySchema 需要擴充
const createTimeEntrySchema = z.object({
  taskId: z.string().nullable(),
  date: z.string(),           // YYYY-MM-DD
  hours: z.number().positive(),
  category: z.nativeEnum(TimeCategory).optional(),
  description: z.string().optional(),
  overtimeType: z.nativeEnum(OvertimeType).optional(),
  subTaskId: z.string().nullable().optional(),
  // v3 新增
  startTime: z.string().optional(),  // HH:mm 格式
  endTime: z.string().optional(),    // HH:mm 格式
}).refine(
  (data) => {
    // 如果有 startTime 就必須有 endTime
    if (data.startTime && !data.endTime) return false;
    if (!data.startTime && data.endTime) return false;
    return true;
  },
  { message: "startTime 和 endTime 必須同時提供" }
);
```

### 第 1 輪：日曆視圖是否過度設計？

**MGR**：我們只有 5 個人在用工時系統。Google Calendar 等級的日曆視圖是不是殺雞用牛刀？我的人光是學怎麼用就要花時間。

**PM**：理解您的擔憂，但這不是功能堆疊。核心問題是：同仁抱怨「格子太像 Excel」。日曆視圖解決的是「直覺性」。5 個人就算只省 5 分鐘/天，一年也是 100+ 小時。

**UX**：而且日曆是所有人都已經會用的介面模式。Google Calendar 和 Apple Calendar 培養出的操作直覺（拖拉、色塊、時段）不需要重新學習。重點是做「簡化版日曆」，不是完整複製 Google Calendar。

**SA**：技術面，日曆視圖的渲染比格子複雜 3-5 倍。但 Schema 不需要改（startTime/endTime 已存在），API 改動小。主要成本在前端元件。

**DEV**：用 CSS Grid 做時間軸其實不難，核心邏輯 < 500 行。拖拉用 @dnd-kit 也成熟。我估計日檢視 3 天、週檢視 5 天、月檢視 3 天可完成 MVP。

**SEC**：從稽核角度，只要 startTime/endTime 的修改都有 audit trail 就沒問題。日曆視圖不增加新的安全風險。

**共識**：日曆視圖不是過度設計，但必須保留格子視圖讓使用者自選。MVP 先做日檢視+週檢視，月檢視第二階段。

### 第 2 輪：起訖時間 vs 時長 — 哪個更實用？

**MGR**：我們是銀行 IT，不是接案公司。很多時候我只需要知道「今天在 A 專案花了 3 小時」，不需要知道是 9-12 還是 14-17。要求填起訖時間會不會增加填寫負擔？

**PM**：這是最關鍵的問題。研究結果顯示：Toggl、Clockify、Tempo 都採用「兩者並存」策略。使用者可以只填時長（在格子裡），也可以填起訖時間（在日曆裡）。我建議 TITAN 也這樣做。

**UX**：起訖時間的好處是「防止灌水」。如果你說「今天做了 8 小時」但每小時都有具體時段，主管更容易驗證。同時，日曆拖拉是最自然的起訖時間輸入方式——不需要手動填兩個時間欄位。

**DEV**：從實作面看，hours 欄位保持 required，startTime/endTime 保持 optional。格子視圖只寫 hours，日曆視圖寫三者。向下相容，沒有 breaking change。

**MGR**：我接受「可選」。但預設模式應該是什麼？我不希望同仁打開工時系統第一眼就是日曆。

**PM**：預設視圖可以由使用者自行設定（localStorage 記住偏好）。第一次進入預設「週格子」（與現有一致），讓想要日曆的人自己切過去。

**SEC**：補充：起訖時間記錄更有利於稽核。銀行勞檢如果問「這個人這天是什麼時候工作的」，有起訖時間才能回答。建議鼓勵但不強制。

**共識**：兩者並存。格子填時長（快速）、日曆填起訖時間（精確）。預設視圖由使用者設定，首次預設為格子。Manager 可透過設定「鼓勵」填起訖時間但不強制。

**DEV**：這估計合理。但有個技術風險：拖拉互動在不同瀏覽器的行為差異。@dnd-kit 處理了大部分，但 Touch 裝置上的 drag 和 scroll 衝突需要額外處理。

**UX**：建議 Phase 1 包含一個精簡版週檢視（只有顯示，不支援拖拉），這樣使用者一開始就能切換日/週看工時分布。

**DEV**：同意。Phase 1 的週檢視可以是「唯讀預覽」+ 點擊日期進入日檢視，拖拉放到 Phase 2。

**SEC**：每個 Phase 交付前都需要 audit trail 測試。

### 第 4 輪：拖拉互動的觸控裝置挑戰

**UX**：手機和平板上的拖拉建立是大問題。手機上「按住拖拉」和「滾動頁面」衝突。Google Calendar 在手機上就不支援拖拉建立，改用「long press → 新增事件」。

**DEV**：確認。@dnd-kit 有 `activationConstraint`，可以設定 long press 閾值（例如 250ms）才啟動拖拉，避免與 scroll 衝突。但這讓拖拉變得不那麼流暢。

**PM**：手機上不需要拖拉建立。主要輸入方式改為：(1) FAB 按鈕快速新增 (2) Long press 時段快速新增 (3) Timer。桌面版才支援拖拉。

**UX**：同意。手機日檢視的區塊互動改為：
- 點擊 → 檢視詳情
- 長按 → 進入編輯模式（可調整時間）
- 左滑 → 刪除確認
- 不支援拖拉建立和 resize

**SA**：用 `@use-gesture/react` 處理手機手勢，與 @dnd-kit 分開。桌面用 @dnd-kit，手機用 @use-gesture，透過 media query 切換。

**MGR**：我們的人主要在辦公室用桌面電腦。手機使用頻率低。桌面體驗優先。

**SEC**：無異議。

**共識**：桌面版支援完整拖拉互動，手機版用 long press + FAB 按鈕替代。觸控平板視螢幕大小決定用哪套互動。

### 第 5 輪：日曆元件的技術選型

**DEV**：市面上有幾個成熟的 React 日曆元件庫：
1. `react-big-calendar` — 功能完整但 bundle 大、樣式客製困難
2. `@fullcalendar/react` — 企業級但 license 複雜
3. `@schedule-x/react` — 新選手，輕量
4. 自建（CSS Grid + DnD Kit）— 完全控制，但開發成本高

我傾向自建，原因：(1) 我們的需求比通用日曆簡單（不需要重複事件、提醒、多日曆）。(2) 套件的客製化成本可能比自建更高。(3) Tailwind 整合更容易。

**SA**：同意自建。核心邏輯分析：
- 時間軸渲染 = CSS Grid，24 行（小時）× 4 列（15 分鐘）= 96 格
- 區塊定位 = absolute positioning，top/height 用百分比
- 拖拉 = @dnd-kit 的 `useDraggable` + `useDroppable`
- 碰撞偵測 = 自建，因為工時不允許重疊

**UX**：自建有個好處是可以做到像素級控制。套件的「通用日曆」外觀很難改成「工時記錄」的語意。我需要控制色塊樣式、tooltip、空白區域的提示文字。

**PM**：自建的風險是什麼？

**DEV**：主要風險：(1) 瀏覽器相容性問題需要自己處理 (2) 無障礙（a11y）需要自己實作 (3) 缺少社群維護。但以 5 人團隊的規模，這些風險可控。

**SEC**：只要前端元件不直接處理資料持久化（都透過 API），安全風險不變。

**共識**：自建日曆元件，使用 CSS Grid + @dnd-kit。核心程式碼控制在 1000 行以內。

### 第 6 輪：時間重疊（Overlap）處理

**UX**：如果使用者拖拉建立的區塊和已有區塊重疊怎麼辦？例如 09:00-12:00 有一筆，又拖了一個 10:00-14:00。

**PM**：真實情況是：一個人在同一時段不可能做兩件事。但工時記錄允許「概略」記錄，不需要 100% 精確。

**DEV**：三個方案：
1. **禁止重疊**：拖拉時自動吸附到不重疊的位置
2. **允許重疊，並排顯示**：像 Google Calendar 同時有兩個會議，並排顯示
3. **允許重疊，提出警告**：可以建立但顯示黃色警示

**MGR**：在實務上，「9-12 開發 + 10-11 臨時被叫去開會」是常見的。開會那 1 小時應該從開發裡扣掉，但很多人懶得改。我傾向「允許但警告」。

**SA**：從資料面看，hours 欄位是獨立的，允許重疊不會造成資料問題。只是「每日總時數」可能超過實際時段的範圍（例如 09:00-17:00 只有 8 小時，但記錄加起來 10 小時）。

**UX**：建議方案：允許重疊 + 視覺提示（重疊區域顯示斜線紋理）+ 日合計超過 8h 時顯示橘色警告。不阻止儲存。

**SEC**：審核時，Manager 應該能看到重疊的記錄並要求修正。在審批介面加上「時間衝突」標記即可。

**共識**：允許時間重疊，並排顯示並加視覺提示（斜線紋理）。日合計超過 8h 顯示橘色警告。Manager 審核介面標記衝突。

---

### 第 7 輪：從現有格子遷移到日曆

**DEV**：現有格子視圖的 TimeEntry 記錄大多沒有 startTime/endTime（只有 hours）。切到日曆視圖時，這些記錄怎麼呈現？

**PM**：好問題。有兩個策略：
1. **智慧推算**：根據 hours 自動推算一個合理的時段（例如 8h → 09:00-17:00）
2. **誠實呈現**：顯示為「未指定時段」的特殊區塊

**UX**：我傾向「誠實呈現」。推算不準會讓使用者困惑。建議設計一個特殊的「浮動區塊」——顯示在日曆底部或頂部的獨立區域，標示為「未指定時段」，使用者可以拖拉它到具體時段來補充起訖時間。

**SA**：技術上，日檢視可以分兩個區域：(1) 上方 = 未指定時段記錄列表（類似全天事件）。(2) 下方 = 時間軸上有起訖時間的記錄。

**MGR**：簡單好用就好。不要強迫我的人回去改舊記錄。新記錄在日曆上填的自然有起訖時間就好。

**DEV**：同意。遷移策略：
1. 舊記錄照顯示，不強制補起訖時間
2. 日曆視圖頂部顯示「未指定時段」區塊
3. 使用者「可選」將舊記錄拖到時間軸上（此時補填 startTime/endTime）
4. 新記錄在日曆上建立時自動有起訖時間

**SEC**：舊記錄不回填符合資料完整性原則。「記錄當時的狀態」比「事後推算」更可靠。

**共識**：舊記錄（無起訖時間）在日曆頂部顯示為「未指定時段」區塊，使用者可選擇性拖入時間軸。不強制回填。

### 第 8 輪：效能考量

**DEV**：日曆視圖的效能瓶頸：
1. 週檢視需要同時渲染 7 天 × 最多 10+ 區塊 = 70+ DOM 元素（含拖拉 handler）
2. 拖拉時需要即時計算位置，每幀更新
3. 月檢視需要載入整月資料

目前 use-timesheet hook 一次只載入一週。需要擴充為按視圖載入。

**SA**：建議：
- 日檢視：載入 1 天的 entries（API 已支援 date filter）
- 週檢視：載入 7 天（現有邏輯）
- 月檢視：載入整月，但只傳 `{ date, totalHours, overtimeHours }` 摘要。點擊日期時再載入該日詳情。

**PM**：5 個人、每人每天平均 3-5 筆記錄，一個月最多 150 筆。不需要分頁或虛擬滾動。

**DEV**：同意。70 個 DOM 元素對 React 不是問題。拖拉效能靠 @dnd-kit 的 `transform`（CSS transform，不觸發 re-render）處理。月檢視用摘要 API 避免一次載入太多詳情。

**UX**：日曆渲染要 < 100ms 才不會讓使用者感到卡頓。建議 memo 化時間區塊元件。

**SEC**：月度摘要 API 需要確保只能看到自己的資料（或 Manager 看所屬成員），不能有 IDOR。

**共識**：5 人規模效能不是問題。月檢視用摘要 API 最佳化。每個 API 都需要權限檢查。

### 第 9 輪：無障礙（Accessibility）

**UX**：日曆拖拉互動對鍵盤使用者和螢幕閱讀器使用者是個挑戰。我們需要考慮 a11y 嗎？

**MGR**：我們 5 個人都沒有無障礙需求。但銀行的 IT 政策可能要求符合 WCAG 2.1 AA。

**SEC**：金管會目前對內部系統沒有強制 WCAG 要求，但建議符合基本標準，避免未來審查風險。

**UX**：建議的 a11y 策略：
1. 所有區塊都有 `aria-label`（例如「TITAN-123 前端重構，09:00 到 12:00，3 小時」）
2. 鍵盤可用 Tab 在區塊間移動，Enter 編輯，Delete 刪除
3. 快速新增（Ctrl+T）是拖拉的鍵盤替代方案
4. 時間選擇器支援鍵盤上下箭頭調整
5. 高對比模式：區塊邊框加粗，不只靠顏色區分

**DEV**：@dnd-kit 原生支援鍵盤拖拉（方向鍵移動）。時間區塊加 `role="button"` 和 `tabindex="0"` 即可。

**PM**：a11y 不增加太多開發成本就做。估計額外 1-2 天。

**共識**：實作基本 a11y（鍵盤操作 + aria-label + 高對比）。不追求完整 WCAG AA。額外成本 1-2 天。

# Part 4：結論與實施建議

## 核心結論

1. **TITAN v3 的定位**：從「填數字的格子」進化為「像行事曆一樣直覺的工時記錄工具」。

4. **分期交付是安全策略**：Phase 1 解決 80% 痛點（日檢視 + 起訖時間），Phase 2/3 看回饋決定。
