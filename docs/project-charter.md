## 1. 專案概述

**TITAN**（Team Integrated Task And kNowledge）是銀行 IT 團隊的現代化協作平台建置專案。

目標是以容器化開源工具取代現行分散的 Email、Excel、紙本文件作業方式，在封閉內網（Air-Gapped）環境中建立統一的任務管理、知識管理、程式碼版控、監控與自動化工作流平台。

---

## 2. MVP 範圍 (Minimum Viable Product)

### 2.1 納入範圍（In Scope）

| 功能模組 | 工具 | 說明 |
|---------|------|------|
| 統一入口 | Homepage | 單一門戶存取所有子系統 |
| 任務管理 | Plane | 看板、甘特圖、Sprint 管理 |
| 知識管理 | Outline | 團隊 Wiki、文件協作 |
| 監控儀表板 | Uptime Kuma | 服務可用性監控與告警 |
| 基礎設施 | PostgreSQL / Redis / MinIO / Nginx | 共用資料層與反向代理 |

> **MVP 階段不含**：Gitea（程式碼版控）、Grafana（指標視覺化）、n8n（工作流自動化）。上述工具列為 Phase C 目標。

### 2.2 排除範圍（Out of Scope）

- 與現有行內核心系統（ERP、HR 系統）的雙向整合
- 行動裝置 App 開發
- 對外網路存取或雲端備份
- 自動化 CI/CD 流水線（Phase C 以後處理）

## 3. 成功標準 (Success Criteria)

| 編號 | 標準 | 驗收方式 | 目標值 |
|------|------|---------|-------|
| SC-01 | 平台可用性 | Uptime Kuma 監控記錄 | ≥ 99.5%（月均） |
| SC-02 | 使用者登入成功率 | AD/LDAP 驗證通過率 | ≥ 99% |
| SC-03 | 任務可追溯性 | Plane 任務完整稽核軌跡 | 100% |
| SC-04 | 文件可搜尋性 | Outline 全文搜尋功能正常 | 搜尋延遲 < 2 秒 |
| SC-05 | 資料不遺失 | 每日備份驗證 | RPO ≤ 24 小時 |
| SC-06 | 回復時間 | 故障回復演練 | RTO ≤ 4 小時 |
| SC-07 | 使用者滿意度 | 上線後 30 天問卷 | ≥ 70% 滿意 |

## 6. 限制條件 (Constraints)

| 類型 | 說明 |
|------|------|
| 網路限制 | 部署環境為完全封閉內網，無法存取外部 Docker Hub 或 apt 來源 |
| 預算限制 | 僅允許使用現有伺服器資源，無額外硬體採購預算 |
| 人力限制 | 團隊 5 人，建置期間需兼顧日常維運工作 |
| 時程限制 | 需於 3 個月內完成核心功能上線 |
| 合規限制 | 必須符合行內資安政策、資料不得外傳 |
| 工具限制 | 僅可使用經資安審核通過的開源工具版本 |

## 7. 假設前提 (Assumptions)

| 編號 | 假設事項 |
|------|---------|
| A-01 | 行內 Active Directory 已建置完成且可供 LDAP 整合使用 |
| A-02 | 已核准使用的開源工具清單在建置期間不會異動 |
| A-03 | 系統管理員可協助配置防火牆規則及 DNS 記錄 |
| A-04 | VM 資源需求（詳見 resource-request.md）已於專案啟動前獲核 |
| A-05 | 所有 Docker 映像已預先下載並轉移至內網 Registry |
| A-06 | 使用者接受在上線初期以使用手冊輔助操作新系統 |
| A-07 | 內部 CA 憑證可供 TLS 使用，且已部署至使用者端 |

## 8. 風險摘要

| 風險 | 可能性 | 影響 | 緩解措施 |
|------|:------:|:----:|---------|
| AD 整合設定複雜導致延誤 | 中 | 高 | 提前取得 LDAP 連線資訊，建立測試環境驗證 |
| Docker 映像轉移不完整 | 低 | 高 | 建立版本清單（version-manifest.md）並驗證 SHA |
| VM 資源申請流程耗時 | 中 | 中 | 提前提交資源需求（resource-request.md）|
| 使用者抗拒新系統 | 中 | 中 | 上線前舉辦說明會，提供操作手冊 |
