# TITAN L1/L2 支援計畫

**文件版本**: v1.0
**撰寫日期**: 2026-03-24
**適用對象**: IT 主管、全體 IT 工程師、使用者
**審核週期**: 每半年複審

---

## 1. 概述

本文件定義 TITAN 工作管理系統的支援架構，包含 L1（一線支援）和 L2（二線支援）的職責劃分、值班安排、升級流程及常見問題處理指引。目標是確保 5 人 IT 團隊能有效分工，在不額外增加人力負擔的前提下維持系統穩定運作。

---

## 2. 支援架構概覽

```
使用者問題回報
      │
      ▼
 L1 支援（值班工程師）
 ● 使用者操作問題
 ● 密碼重設
 ● 簡單 FAQ 問題
      │
      │ 若 L1 無法解決（30 分鐘內）
      ▼
 L2 支援（資深工程師）
 ● 系統異常排查
 ● 效能問題分析
 ● 設定變更
 ● 資料修復
      │
      │ 若涉及架構或安全問題
      ▼
 IT 主管（最終決策）
 ● 架構調整
 ● 資安事件處理
 ● 對外協調
```

## 3. L1 支援範圍

### 3.1 職責定義

L1 支援由值班工程師負責，處理使用者可直接觀察的問題，不需深入系統診斷。

**L1 處理範圍**：

| 問題類型 | 具體範例 | 處理方式 |
|---------|---------|---------|
| 帳號存取問題 | 無法登入、帳號被鎖定 | 依 FAQ 操作解鎖或重設 |
| 密碼重設 | 忘記密碼 | 使用管理員功能重設 |
| 操作疑問 | 如何建立任務、如何指派負責人 | 提供教學或導向使用者手冊 |
| 介面顯示問題 | 頁面空白、圖示不顯示 | 請使用者清快取、換瀏覽器 |
| 基本資料維護 | 更新個人資料、偏好設定 | 操作指引 |
| 通知問題 | 未收到系統通知 | 確認通知設定，基本排查 |

**L1 不處理範圍**（升級 L2）：

- 系統持續無法使用（超過 5 分鐘）
- 資料異常（數字錯誤、紀錄消失）
- 效能明顯降低（頁面載入超過 10 秒）
- 錯誤訊息包含「500 Internal Server Error」或「502 Bad Gateway」
- 任何涉及資安疑慮的問題

### 3.2 帳號解鎖操作步驟

```
1. 以管理員身分登入 TITAN
2. 進入 [設定] > [使用者管理]
3. 找到被鎖定帳號（狀態顯示「鎖定」）
4. 點選 [解鎖帳號]
5. 確認使用者是本人申請（電話或面對面確認）
6. 告知使用者已解鎖，建議變更密碼
7. 在值班日誌記錄操作內容
```

### 3.3 密碼重設操作步驟

```
1. 確認使用者身分（工號、直屬主管確認）
2. 以管理員身分進入 [設定] > [使用者管理]
3. 找到對應帳號，點選 [重設密碼]
4. 系統產生臨時密碼
5. 將臨時密碼透過安全管道告知使用者（面對面或內部電話，不可 Email）
6. 告知使用者首次登入後必須立即變更密碼
7. 記錄操作於值班日誌
```

## 4. L2 支援範圍

### 4.1 職責定義

L2 支援由資深工程師負責，需要系統層面的診斷能力，包含查看日誌、調整設定、分析效能。

**L2 處理範圍**：

| 問題類型 | 具體範例 | 工具/方法 |
|---------|---------|---------|
| 系統服務異常 | 應用容器崩潰、資料庫連線失敗 | Docker logs、docker-compose 重啟 |
| 效能問題 | API 回應慢、頁面長時間載入 | Grafana Dashboard、PostgreSQL EXPLAIN |
| 資料異常排查 | 特定任務資料不正確 | 直接查詢資料庫驗證 |
| 設定變更 | 修改環境變數、Nginx 設定 | 設定檔編輯 + 服務重載 |
| 日誌分析 | 調查特定時間段的異常請求 | 系統日誌分析 |
| 備份/復原操作 | 執行緊急備份、資料還原 | 依 disaster-recovery.md 操作 |
| 依賴套件更新 | 安全漏洞修補 | npm audit fix、Docker image 更新 |

**L2 升級 IT 主管的情況**：

- 問題超過 1 小時無法解決
- 需要修改核心架構
- 懷疑發生資安事件
- 需要執行破壞性操作（如資料庫清除）
- 影響範圍超出 TITAN 系統（涉及其他系統或網路設備）

### 4.2 常用診斷指令

**確認系統狀態**

```bash
# 查看所有容器狀態
cd /opt/titan && docker-compose ps

# 查看應用日誌（最近 100 行）
docker-compose logs --tail=100 app

# 查看資料庫日誌
docker-compose logs --tail=100 postgres

# 查看即時日誌（持續監看）
docker-compose logs -f app
```

**效能診斷**

```bash
# 查看容器資源使用
docker stats --no-stream

# 查看 PostgreSQL 慢查詢（>1 秒）
docker-compose exec postgres psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  -c "SELECT query, mean_exec_time, calls FROM pg_stat_statements
      WHERE mean_exec_time > 1000 ORDER BY mean_exec_time DESC LIMIT 10;"

# 查看資料庫連線數
docker-compose exec postgres psql -U "$POSTGRES_USER" -d "$POSTGRES_DB" \
  -c "SELECT count(*) FROM pg_stat_activity;"
```

**服務重啟**

```bash
# 重啟應用（不重啟資料庫）
docker-compose restart app

# 重啟所有服務
docker-compose down && docker-compose up -d

# 強制重建應用容器
docker-compose up -d --force-recreate app
```

## 5. 值班安排建議

### 5.1 5 人團隊輪值方案

TITAN 為內部系統，建議採用**輕量值班制**，不需 24 小時全日值班，重點確保核心服務時間（週一至週五 08:00-19:00）有人響應。

**週值班輪替方案**：

| 週次 | L1 值班 | L2 備援 |
|------|--------|--------|
| 第 1 週 | 工程師 A | 工程師 E（資深）|
| 第 2 週 | 工程師 B | 工程師 D（資深）|
| 第 3 週 | 工程師 C | 工程師 E（資深）|
| 第 4 週 | 工程師 D | 工程師 E（資深）|
| 第 5 週（如有）| 工程師 A | 工程師 D（資深）|

> IT 主管不排入 L1/L2 輪值，但保持可聯繫狀態，負責升級事件的最終決策。

### 5.2 值班職責

**L1 值班工程師（每週輪替）**

- 每日 08:00 確認系統 Health Check 正常
- 接收並分類使用者問題回報
- 30 分鐘內回應 L1 問題
- 超出 L1 範圍立即通知 L2 備援

**L2 值班備援（每週輪替，優先由資深工程師擔任）**

- 接收 L1 升級的問題
- 1 小時內回應
- 超出 L2 範圍立即通知 IT 主管

**IT 主管**

- 保持手機可聯繫（核心服務時間）
- P1/P2 事件的最終決策人
- 每月審閱值班日誌與事件報告

### 5.3 問題回報管道

| 問題類型 | 回報管道 | 備註 |
|---------|---------|------|
| 一般操作問題 | TITAN 系統內建 Help Desk 模組 | 首選管道 |
| 緊急問題（系統無法使用）| 直接電話/即時通訊聯絡值班工程師 | P1/P2 適用 |
| 非緊急問題 | Email 至 IT 部門信箱 | 預計 4 小時內回覆 |

## 6. 常見問題 FAQ

### Q1: 登入時顯示「帳號已鎖定」

**原因**：連續輸入錯誤密碼 5 次後，系統自動鎖定帳號。

**解決方式**：
- 等待 15 分鐘後系統自動解鎖，再次嘗試登入
- 若需立即解鎖，聯絡 L1 值班工程師（提供工號驗證身分）

---

### Q2: 忘記密碼，無法登入

**解決方式**：
- 聯絡 L1 值班工程師，確認身分後由管理員重設臨時密碼
- 收到臨時密碼後，首次登入時系統會強制要求變更密碼
- 注意：不接受 Email 申請密碼重設，需本人確認

---

### Q3: 頁面顯示空白或載入一直轉圈

**先嘗試以下步驟**：
1. 按 `Ctrl+F5`（Windows）或 `Cmd+Shift+R`（Mac）強制重新整理
2. 清除瀏覽器快取（設定 → 清除瀏覽資料）
3. 換一個瀏覽器試試（建議使用 Chrome 或 Firefox）
4. 確認網路連線正常

**若以上步驟無效**：聯絡 L1 值班工程師，告知具體頁面名稱和出現的錯誤訊息。

---

### Q4: 任務建立後，其他人看不到

**可能原因**：
- 任務被設定為私人（Private）模式
- 任務指派的專案/群組其他人沒有存取權限

**解決步驟**：
1. 進入任務詳情，確認「可見度」設定
2. 確認任務所在專案的成員設定
3. 若設定正確但問題持續，聯絡 L1 值班工程師

---

### Q5: 系統回應很慢，操作要等很久

**先確認**：
- 是個人網路問題還是所有人都慢？（詢問同事）
- 是特定頁面慢還是整個系統都慢？

**若只有自己慢**：可能是個人端網路或電腦問題，請 IT 協助排查個人端。

**若所有人都慢**：聯絡 L1 值班工程師，描述：開始變慢的時間點、哪個頁面最明顯。

---

### Q6: 不小心刪除了任務，能還原嗎？

**軟刪除機制**：TITAN 刪除操作為軟刪除，資料不會立即從資料庫移除。

**還原方式**：
- 管理員可在 [設定] > [已刪除任務] 中找到並還原（保留 30 天）
- 30 天後資料永久刪除，無法還原
- 聯絡 L1/L2 工程師協助還原

---

### Q7: 系統顯示「500 Internal Server Error」

這是系統內部錯誤，請立即聯絡 L1 值班工程師，提供：
- 出現錯誤的時間
- 執行的操作（點了哪個按鈕、提交什麼表單）
- 瀏覽器顯示的完整錯誤訊息

---

### Q8: 如何匯出報表或工作記錄？

1. 進入 [報表] 功能
2. 選擇時間範圍和篩選條件
3. 點選 [匯出 CSV] 或 [匯出 Excel]
4. 若需要特定格式報表，聯絡 IT 工程師評估開發

## 7. 監控告警處理流程

### 7.1 自動告警觸發條件

TITAN 的 Prometheus + Grafana 監控系統設定以下告警：

| 告警名稱 | 觸發條件 | 嚴重度 | 通知對象 |
|---------|---------|--------|---------|
| ServiceDown | 應用服務無回應超過 2 分鐘 | Critical | L1 + L2 + IT 主管 |
| DatabaseDown | 資料庫連線失敗超過 1 分鐘 | Critical | L1 + L2 + IT 主管 |
| HighResponseTime | API P95 回應時間 > 2 秒持續 5 分鐘 | Warning | L1 + L2 |
| HighCpuUsage | CPU 使用率 > 80% 持續 10 分鐘 | Warning | L2 |
| HighMemoryUsage | 記憶體使用率 > 85% | Warning | L2 |
| DiskSpaceWarning | 磁碟使用率 > 80% | Warning | L2 |
| BackupFailed | 備份作業失敗 | Warning | L1 + IT 主管 |
| LoginFailureSpike | 5 分鐘內登入失敗超過 20 次 | Warning | L2 + IT 主管 |

### 7.2 告警收到後的標準處理流程

```
收到告警通知
      │
      ▼
1. 確認告警真實性（非誤報）
   - 登入 Grafana 查看詳細指標
   - 嘗試自行存取 TITAN 確認
      │
      ▼
2. 判斷影響範圍
   - 影響所有人 → P1/P2
   - 影響部分使用者 → P3
      │
      ▼
3. 按 SLA 定義的 RTO 開始處理
   - Critical 告警：立即開始，15 分鐘內回報狀態
   - Warning 告警：1 小時內評估
      │
      ▼
4. 執行對應的排障步驟
   （參考本文件 L2 診斷指令或 disaster-recovery.md）
      │
      ▼
5. 問題解決後
   - 填寫事件記錄（TITAN 系統）
   - 通知相關使用者服務恢復
   - P1/P2 需填寫 RCA 報告（24 小時內）
```

### 7.3 告警靜音管理

- 計劃維護期間，L2 工程師可暫時靜音相關告警（最長 4 小時）
- 靜音前須通知 IT 主管
- 靜音期間若發生真實事件，L2 工程師需主動確認

## 8. 值班日誌規範

每位值班工程師在值班期間須維護值班日誌（記錄於 TITAN 系統），內容包含：

**日誌格式**：

```
日期：YYYY-MM-DD
值班工程師：[姓名]
系統狀態：正常 / 有問題（說明）

事件記錄：
[時間] [問題描述] [處理方式] [結果] [是否升級]

例：
09:15  使用者 A 反映無法登入（帳號鎖定）  → 驗證身分後手動解鎖  → 已解決
14:30  API 回應時間告警觸發（P95 = 2.5s） → 查詢為批次報表導致  → 已優化查詢，恢復正常
```
