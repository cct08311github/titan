## 1. 設計原則

TITAN 知識庫遵循以下設計原則：

| 原則         | 說明                                                        |
|--------------|-------------------------------------------------------------|
| 單一來源原則 | 每份文件只存在一個位置，避免重複維護                        |
| 搜尋優先     | 文件標題與內容需包含關鍵詞，方便全文搜尋                    |
| 範本驅動     | 所有文件類型均有對應範本，降低撰寫門檻                      |
| 漸進更新     | 文件「足夠好」即可發布，持續迭代優於延遲發布               |
| 可追溯性     | 重大決策記錄於 Architecture 空間，包含背景、考量與決定      |

## 2. 空間（Spaces）架構

TITAN Outline 規劃五個一級空間，對應不同文件類型與使用族群：

```
TITAN Workspace
├── SOP（標準作業程序）        — 操作手冊、步驟指引
├── Runbook（維運手冊）        — 故障排除、事故處理
├── FAQ（常見問答）            — 快速解答、疑難排解
├── Meeting Notes（會議記錄） — 會議紀要、決議追蹤
└── Architecture（架構設計）  — 技術決策、系統設計
```

### 空間摘要

| 空間名稱       | 主要讀者       | 主要寫者           | 更新頻率 |
|----------------|----------------|--------------------|----------|
| SOP            | 所有 IT 成員   | 程序負責人 / PM    | 低（程序變更時）|
| Runbook        | 維運工程師     | DevOps / 工程師    | 中（每季回顧）  |
| FAQ            | 所有成員       | 任何人             | 高（隨時新增）  |
| Meeting Notes  | 與會成員       | 會議記錄者         | 高（每次會議後）|
| Architecture   | 技術成員       | 架構師 / 主任工程師| 低（決策時）    |

## 3. 各空間詳細規劃

### 3.1 SOP（標準作業程序）

**目的**：記錄可重複執行的標準程序，確保任何人都能依照步驟完成任務。

**建議子集合結構**：

```
SOP/
├── 部署作業/
│   ├── 服務部署 SOP
│   ├── 資料庫 Schema 更新 SOP
│   ├── Docker 映像發布 SOP
│   └── 回滾作業 SOP
├── 帳號管理/
│   ├── 新成員入職 SOP
│   ├── 帳號停用 SOP
│   └── 權限申請 SOP
├── 備份與復原/
│   ├── 每日備份驗證 SOP
│   └── 資料復原 SOP
└── 安全管理/
    ├── 憑證更新 SOP
    ├── 漏洞修補 SOP
    └── 安全事件通報 SOP
```

**使用範本**：`templates/outline/sop-template.md`

**品質標準**：
- 每個步驟必須可驗證（有明確輸出或成功條件）
- 包含前置條件、注意事項、緊急聯絡人
- 每年至少審查一次，重大程序變更時立即更新

---

### 3.2 Runbook（維運手冊）

**目的**：記錄服務維運知識，包含健康檢查、常見問題排除、事故處理程序。

**建議子集合結構**：

```
Runbook/
├── 服務維運/
│   ├── Homepage 維運手冊
│   ├── Plane 維運手冊
│   ├── Outline 維運手冊
│   ├── Gitea 維運手冊
│   └── Harbor 維運手冊
├── 基礎設施/
│   ├── PostgreSQL 維運手冊
│   ├── Redis 維運手冊
│   ├── MinIO 維運手冊
│   └── Nginx 維運手冊
├── 監控告警/
│   ├── 告警規則說明
│   ├── 常見告警處理指引
│   └── Grafana 儀表板說明
└── 事故管理/
    ├── P0 事故處理流程
    ├── 事故後 RCA 範本
    └── 歷史事故記錄
```

**使用範本**：`templates/outline/runbook-template.md`

**品質標準**：
- 每個告警對應至少一條排除步驟
- 包含服務依賴關係圖
- 每季 Runbook Review，確保步驟與現況一致

---

### 3.3 FAQ（常見問答）

**目的**：快速解答成員常見問題，降低重複詢問，加速新成員上手。

**建議子集合結構**：

```
FAQ/
├── 入職 FAQ/
│   ├── 環境設定常見問題
│   ├── 帳號申請流程
│   └── 工具使用入門
├── 開發工具 FAQ/
│   ├── Gitea 使用問題
│   ├── Harbor 映像倉庫問題
│   └── CI/CD 流程問題
├── 服務存取 FAQ/
│   ├── VPN 連線問題
│   ├── SSO 登入問題
│   └── 權限申請流程
└── 維運 FAQ/
    ├── 服務重啟步驟
    ├── 日誌查看方式
    └── 備份還原問題
```

**使用範本**：`templates/outline/faq-template.md`

**品質標準**：
- 問答格式簡潔（問題 → 解答 → 相關連結）
- 新問題出現兩次以上即應新增至 FAQ
- 解答過時時標記 `⚠️ 待更新` 並指派負責人

---

### 3.4 Meeting Notes（會議記錄）

**目的**：記錄會議決議、行動項目，確保決策可追溯且行動可追蹤。

**建議子集合結構**：

```
Meeting Notes/
├── Sprint 會議/
│   ├── Sprint Planning（每 Sprint 一份）
│   ├── Sprint Review（每 Sprint 一份）
│   └── Sprint Retrospective（每 Sprint 一份）
├── 技術討論/
│   ├── 架構評審會議
│   └── 技術選型討論
├── 管理會議/
│   ├── 週會記錄
│   └── 里程碑回顧
└── 事故檢討/
    └── RCA 會議記錄
```

**使用範本**：`templates/outline/meeting-notes-template.md`

**命名規範**：`YYYY-MM-DD 會議名稱`（例：`2026-03-23 Sprint 5 Planning`）

**品質標準**：
- 會議結束後 24 小時內完成記錄
- 每個行動項目（Action Item）指定負責人與截止日期
- 行動項目同步建立至 Plane Issue

---

### 3.5 Architecture（架構設計）

**目的**：記錄技術決策（ADR）、系統架構設計、整合規格，作為長期技術參考。

**建議子集合結構**：

```
Architecture/
├── 決策紀錄（ADR）/
│   ├── ADR-001 選用 Plane 作為任務管理工具
│   ├── ADR-002 Keycloak 作為統一身份驗證
│   ├── ADR-003 MinIO 作為物件儲存
│   └── ADR-NNN ...
├── 系統架構/
│   ├── TITAN 整體架構圖
│   ├── 網路拓樸設計
│   ├── 服務依賴關係圖
│   └── 資料流程圖
├── 整合規格/
│   ├── SSO / OIDC 整合規格
│   ├── API Gateway 規格
│   └── 服務間通訊規格
└── 安全架構/
    ├── 安全控制清單
    ├── 資料分類與處理標準
    └── 稽核日誌設計
```

**使用範本**：`templates/outline/cr-template.md`（架構變更請求）

**命名規範**：ADR 使用流水號 `ADR-NNN 標題`

**品質標準**：
- ADR 必須包含背景、考量的替代方案、決定理由
- 架構圖以 Mermaid 或 Draw.io 嵌入，版本與文件同步更新
- 重大架構變更需通過架構評審會議核准

## 4. 範本（Templates）清單

所有範本存放於 `templates/outline/`，與 T13 同步產出。

| 範本檔案                   | 用途                         | 對應空間         |
|----------------------------|------------------------------|------------------|
| `sop-template.md`          | 標準作業程序                 | SOP              |
| `runbook-template.md`      | 服務維運手冊                 | Runbook          |
| `faq-template.md`          | 常見問題解答                 | FAQ              |
| `meeting-notes-template.md`| 會議記錄                     | Meeting Notes    |
| `cr-template.md`           | 架構變更請求 / ADR           | Architecture     |

### 在 Outline 中匯入範本步驟

1. 進入目標空間（Collection）
2. 點擊 **New document** → **From template**
3. 若範本尚未建立：複製 `templates/outline/` 中的對應 Markdown 內容建立範本文件
4. 在空間設定中將文件標記為 Template

## 5. 文件命名規範

| 文件類型     | 命名格式                           | 範例                              |
|--------------|------------------------------------|-----------------------------------|
| SOP          | `[服務/流程名稱] SOP`              | `PostgreSQL 備份 SOP`             |
| Runbook      | `[服務名稱] 維運手冊`              | `Plane 維運手冊`                  |
| FAQ          | `[主題] FAQ`                       | `Gitea 使用問題 FAQ`              |
| 會議記錄     | `YYYY-MM-DD [會議名稱]`            | `2026-03-23 Sprint 5 Planning`    |
| ADR          | `ADR-NNN [決策標題]`               | `ADR-003 選用 MinIO 作為物件儲存` |
| 架構文件     | `[系統名稱] 架構設計`              | `TITAN 整體架構設計`              |

**通用規則**：
- 使用繁體中文標題
- 避免特殊字元（除連字號 `-` 和底線 `_`）
- 標題應能獨立說明文件內容（不需看內容即知用途）

## 6. 標籤（Tags）系統

Outline 支援文件標籤，建議建立以下標籤：

| 標籤              | 說明                            |
|-------------------|---------------------------------|
| `待更新`          | 內容可能過時，需確認或更新      |
| `草稿`            | 尚未完成，僅供參考              |
| `已審查`          | 經過正式審查，可信度高          |
| `機密`            | 含敏感資訊，限制存取            |
| `v1` / `v2`       | 版本標記（適用於有版本的文件）  |

## 7. 權限設計

| 空間           | 可讀取        | 可編輯                  | 說明                         |
|----------------|---------------|-------------------------|------------------------------|
| SOP            | 所有成員      | PM、程序負責人          | 保持穩定，避免隨意修改       |
| Runbook        | 所有成員      | DevOps、工程師          | 鼓勵工程師持續更新           |
| FAQ            | 所有成員      | 所有成員                | 開放協作，任何人可新增問答   |
| Meeting Notes  | 相關成員      | 會議記錄者              | 部分會議可設為私有           |
| Architecture   | 技術成員      | 架構師、主任工程師      | 嚴格控管，避免非授權修改     |

### Keycloak 群組對應

| Outline 權限群組 | Keycloak 群組         |
|------------------|-----------------------|
| Admin            | `titan-admins`        |
| Editor           | `titan-engineers`     |
| Viewer           | `titan-readonly`      |

## 8. 初始建置步驟

### 前置條件

- Outline 已部署並可存取（參考 `docs/outline-setup.md`）
- Keycloak OIDC 整合已完成（參考 `docs/auth-design.md`）
- 管理員帳號已建立

### 建置清單

- [ ] 建立 Workspace：`TITAN`
- [ ] 設定 OIDC 整合（Keycloak）
- [ ] 建立五個一級空間：SOP、Runbook、FAQ、Meeting Notes、Architecture
- [ ] 各空間建立建議子集合結構（依 [第 3 節](#3-各空間詳細規劃)）
- [ ] 匯入 `templates/outline/` 中的 5 個範本
- [ ] 建立標籤系統（依 [第 6 節](#6-標籤tags系統)）
- [ ] 設定各空間權限（依 [第 7 節](#7-權限設計)）
- [ ] 建立 SOP 空間的第一份文件：服務部署 SOP
- [ ] 建立 Architecture 空間的第一份 ADR：TITAN 技術選型決策

### 新成員入職說明

新成員加入時，導覽以下資源：

1. **FAQ/入職 FAQ** — 環境設定與帳號申請
2. **SOP/帳號管理/新成員入職 SOP** — 詳細入職流程
3. **Architecture/系統架構/TITAN 整體架構圖** — 了解整體系統
