# 銀行 IT 團隊 MVP 細部規劃與工作任務拆解

## 0. 專案背景與目標

### 0.1 背景
- 團隊規模：5 人工程師團隊
- 環境限制：封閉網路、不可連外、需符合銀行內控與資安要求
- MVP 目標：在內網建立一套可用、可維運、可審計的協作平台，整合：
  - **Plane**：任務管理
  - **Outline**：知識庫 / 文件中心
  - **Homepage**：統一入口 / 導航頁

### 0.2 MVP 核心目標
1. 讓團隊先有**單一登入入口與單一入口頁**
2. 讓專案任務能在內網中完成建立、追蹤、指派、看板管理
3. 讓 SOP、系統文件、維運手冊能集中存放與搜尋
4. 讓整套服務能在封閉網路中穩定部署、備份、還原與稽核

### 0.3 設計原則
- **先求可用，再求完整**：優先完成最小可行流程
- **先共用底座，再建業務功能**：優先處理登入、權限、入口、備份、監控
- **先內網穩定，再擴功能**：先不做外部整合、通知串接、複雜自動化
- **先審計合規，再談效率優化**：所有部署與變更需可追溯

# 1. MVP 範圍定義

## 1.1 範圍總覽

### P0：MVP 必做（第一階段必上線）
這些功能是平台可交付的最低條件。

#### 共用平台能力
- 內網 DNS / URL 規劃
- 反向代理（Nginx / Traefik，建議 Nginx）
- 企業 AD / LDAP 或本機帳號整合（若 AD 尚未開放，先以內部帳號機制替代）
- HTTPS 憑證（內部 CA）
- RBAC 權限模型
- 備份與還原機制
- 日誌留存與存取審計
- 監控與服務健康檢查

#### Homepage
- 統一入口頁
- Plane / Outline / 常用系統連結卡片
- 服務狀態顯示（可簡化為連通檢查）
- 依角色顯示不同入口

#### Plane
- 建立工作區 / 專案
- 任務建立、狀態流轉、指派、截止日、標籤
- Kanban 看板
- 基本搜尋 / 篩選
- 使用者與群組權限

#### Outline
- 文件空間 / 分類
- 頁面建立、編輯、版本紀錄
- 全文搜尋
- 權限控管（依部門 / 群組）
- SOP / Runbook / FAQ 模板

### P1：建議第二階段（MVP 後 1~2 個 Sprint）
- Plane 與 Outline 的交叉連結規範（例如任務可附文件）
- Homepage 顯示待辦摘要 / 常用知識文章
- 通知機制（郵件或內部訊息，若封閉網路有 SMTP）
- 文件模板標準化與審核流程
- 任務報表（人力負載、延期統計）
- SSO 正式接入（若 MVP 先以 local auth）

### P2：後續擴充（非 MVP）
- 行動裝置優化
- API Gateway 與系統整合
- CMDB / ITSM / GitLab / CI 串接
- 自動化知識推薦
- 細粒度審批流 / 電子簽核
- HA 雙機 / 跨機房容災

## 1.2 先做 / 後做建議

### 先做
1. 基礎環境與內網部署底座
2. Homepage 單一入口
3. Plane 核心任務管理
4. Outline 核心知識庫
5. 帳號權限、備份、稽核、監控
6. UAT 與上線文件

### 後做
1. 進階報表
2. 進階通知
3. 系統間深度整合
4. 高可用與跨站容災
5. 自動化流程與 API 擴充

## 1.3 不納入 MVP 的項目
- 與外網 SaaS 整合
- 即時聊天 / Bot 通知
- 複雜流程引擎
- 大量自訂開發插件
- 全公司級推廣流程
- 多機房同步備援

# 2. 團隊角色與分工

## 2.1 建議角色配置（5 人）

| 角色 | 人數 | 代號 | 主要職責 |
|---|---:|---|---|
| 專案/技術負責人 | 1 | TL | 排程、架構決策、跨組溝通、風險控管 |
| 平台/DevOps 工程師 | 1 | DevOps | 伺服器、容器、反向代理、部署、監控、備份 |
| 後端/整合工程師 | 1 | Backend | Plane/Outline/Homepage 整合、帳權與 API 配置 |
| 前端/入口工程師 | 1 | Frontend | Homepage 客製化、UI 導覽、入口整合 |
| 測試/維運工程師 | 1 | QA/Ops | 測試案例、UAT、操作手冊、上線驗證、稽核證據 |

> 若實際上 5 人皆為全端工程師，仍建議依責任主軸分工，避免無主項目。

## 2.2 RACI 簡化原則
- **TL**：對所有交付物最終 accountable
- **DevOps**：環境類項目 responsible
- **Backend / Frontend**：應用層 responsible
- **QA/Ops**：驗證、文件、上線檢查 responsible

## 3.2 任務分群與說明

### A. 規劃與基礎準備
- T01~T05
- 目標：先把需求、拓樸、資源、版本來源定義清楚
- 風險：封閉網路環境最怕「安裝時才發現缺套件或權限」

### B. 基礎設施與安全底座
- T06~T10
- 目標：先打好 OS、容器、TLS、帳權、儲存底座
- 風險：AD/LDAP、內部 CA、DNS 任何一項未就緒都會拖延

### C. 應用建置與整合
- T11~T17
- 目標：將三套服務整合成可實際使用的平台
- 風險：版本相容性、資料結構、登入權限模型不一致

### D. 維運與合規
- T18~T20
- 目標：補齊監控、備份、審計、權限驗證
- 風險：若最後才補，通常會造成延誤與重工

### E. 測試、上線與收斂
- T21~T26
- 目標：系統整體驗證、上線與營運接手
- 風險：UAT 若沒有標準情境，會反覆修改範圍

# 5. 技術規格

## 5.1 部署模式建議

### MVP 建議：單一應用區 + 獨立資料備份區
考量 5 人團隊與 MVP 速度，建議先採：
- **1 台應用主機**：跑 Nginx、Homepage、Plane、Outline
- **1 台資料與備份主機**：跑 PostgreSQL、Redis、備份檔、還原驗證
- 若資源有限，也可先單機，但正式環境不建議所有元件共機

## 5.2 伺服器規格建議

### 正式環境（最小建議）

#### APP-01（應用主機）
- vCPU：8 Core
- RAM：32 GB
- Disk：500 GB SSD
- OS：Rocky Linux / RHEL / Ubuntu LTS（依銀行標準）
- 用途：Nginx、Homepage、Plane、Outline、應用日誌

#### DB-01（資料與備份主機）
- vCPU：8 Core
- RAM：32 GB
- Disk：1 TB SSD
- 用途：PostgreSQL、Redis、附件儲存、每日備份檔、還原測試區

### 測試 / UAT 環境
- vCPU：4 Core
- RAM：16 GB
- Disk：250~500 GB
- 可採共機，但需與正式隔離

## 5.3 軟體元件建議

| 元件 | 建議技術 |
|---|---|
| OS | RHEL/Rocky/Ubuntu LTS |
| 反向代理 | Nginx |
| 容器執行 | Docker Engine 或 Podman |
| 編排 | docker compose |
| DB | PostgreSQL |
| Cache / Queue | Redis |
| 備份 | pg_dump + rsync / restic（若內網允許） |
| 監控 | Prometheus + Grafana / 或既有內部監控 |
| 日誌 | OS syslog + 應用 log rotate + SIEM 匯入 |

## 5.4 網路配置規劃

### 網段與區域建議
- **管理區**：僅管理者可 SSH / RDP 連入
- **應用區**：使用者可存取 Homepage / Plane / Outline
- **資料區**：僅 APP-01 可連 DB-01 指定 Port
- **備份區**：僅備份程序可寫入

### Port 規劃示例
| 服務 | Port | 存取來源 | 備註 |
|---|---:|---|---|
| HTTPS / Nginx | 443 | 內網使用者 | 對外唯一入口 |
| HTTP | 80 | 僅轉導 | 強制跳轉 HTTPS |
| PostgreSQL | 5432 | 僅 APP-01 | 防火牆白名單 |
| Redis | 6379 | 僅 APP-01 | 不對使用者開放 |
| SSH | 22 | 管理區跳板機 | 嚴禁全網開放 |

### DNS / FQDN 建議
- `portal.bank.intra` → Homepage
- `task.bank.intra` → Plane（也可透過 portal 導向）
- `wiki.bank.intra` → Outline（也可透過 portal 導向）

> 若要降低複雜度，也可全部掛在同一網域下的不同路徑，但實務上用子網域較清楚。

## 5.5 帳號與權限規劃

### 角色模型
1. **Platform Admin**
   - 管理系統設定、使用者、群組、備份、監控
2. **Team Manager**
   - 可建立/管理專案、查看報表、維護部門知識空間
3. **Engineer / Contributor**
   - 建立/更新任務、撰寫知識文件、查閱共用文件
4. **Viewer / Auditor**
   - 僅查閱、不可修改，適合稽核與管理層

### 帳號來源策略
- 優先：AD / LDAP 群組對映
- 次選：系統 local account + 定期審核

### 權限原則
- 最小權限原則（Least Privilege）
- 管理權限與一般使用權限分離
- 正式環境不得共用帳號
- 所有管理行為需可追溯
- 離職 / 調職需有權限回收流程

# 6. 開發與維運規範（銀行業）

## 6.1 程式碼規範
- 所有設定與客製化腳本納入 Git 版本控管
- 分支策略：`main`、`release/*`、`feature/*`、`hotfix/*`
- Commit message 標準化：`feat:`、`fix:`、`chore:`、`docs:`、`sec:`
- 不可將密碼、Token、憑證私鑰寫入 repo
- 所有環境變數統一由安全設定檔管理
- 客製化程式碼必須附：
  - 用途說明
  - 依賴版本
  - 部署方式
  - 回滾方式
- 第三方映像 / 套件需保留版本凍結清單與 checksum

## 6.2 部署流程

### 環境分層
- DEV / SIT / UAT / PROD 至少邏輯隔離
- 正式環境變更需有申請、審核、執行、驗證、回滾紀錄

### 標準部署流程
1. 開發完成
2. Code Review 通過
3. 安全掃描 / 套件檢查通過
4. SIT 部署驗證
5. UAT 驗證與簽核
6. 正式變更單申請
7. 上線窗口執行
8. 上線後驗證
9. 觀察期結束後結案

### 上線必備文件
- 部署 SOP
- 參數表
- Port / DNS 清單
- 驗證清單
- 回滾 Runbook
- 備份確認紀錄
- 緊急聯絡表

## 6.3 備份策略

### 備份範圍
- PostgreSQL 資料庫
- Outline 附件 / 文件資產
- Plane 設定與資料
- Homepage 設定檔
- Nginx / Compose / 環境變數設定檔
- TLS 憑證與續期資料（若適用）

### 備份頻率建議
- DB：每日全備 + 每 4 小時增量/快照（若工具支援）
- 附件檔：每日備份
- 設定檔：每次變更後即備份
- 備份保存：
  - 日備份保留 7~14 天
  - 週備份保留 8 週
  - 月備份保留 6~12 個月（依內規）

### 還原策略
- 每月至少一次還原演練
- 還原驗證需包含：
  - DB 可啟動
  - 登入可用
  - 文件與附件可讀取
  - 任務資料完整
- 備份檔需校驗 checksum
- 備份儲存區不可被一般使用者直接讀寫

## 6.4 資安檢核

### 主機層
- 關閉不必要服務與 Port
- 套用 OS 安全更新
- 禁止 root 直接遠端登入
- SSH 僅允許跳板機與指定管理 IP
- 使用 sudo 審計
- 啟用檔案權限最小化

### 應用層
- 全站 HTTPS
- 安全 Header
- Session timeout 設定
- 強密碼政策 / AD 密碼政策沿用
- 預設帳號關閉或改密
- 不使用預設密鑰
- 管理介面不得暴露於一般網段

### 供應鏈層
- 第三方映像需經掃描後匯入內部 registry
- 嚴禁正式環境直接 pull 外部映像
- 套件來源需可追溯
- 保留 SBOM（Software Bill of Materials）更佳

### 稽核層
- 登入成功 / 失敗記錄
- 權限變更記錄
- 設定變更記錄
- 備份 / 還原記錄
- 上線與回滾記錄

# 7. 驗收標準

## 7.1 MVP 功能驗收
- 使用者可透過 Homepage 進入 Plane 與 Outline
- 管理者可建立使用者/群組或完成 AD 群組對映
- Plane 可建立專案、任務、看板並完成指派與狀態更新

- 監控頁可看到主要服務狀態

## 7.2 非功能驗收
- 內網登入與操作延遲在可接受範圍內
- 所有服務以 HTTPS 提供
- 重大弱點掃描無 High/Critical 未處理項
- 具完整上線文件、操作手冊、回滾手冊

# 8. 風險評估與解法

## 8.1 風險總表

| 風險 | 說明 | 影響 | 解法 |
|---|---|---|---|
| 離線安裝包不完整 | 封閉網路無法臨時下載缺件 | 高 | Week 1 完成版本凍結、映像匯入、checksum 驗證 |
| AD/LDAP 串接延誤 | 權限與登入無法按時完成 | 高 | MVP 預備 local auth 備援方案，P1 再切 LDAP |
| Plane/Outline 版本相容問題 | 部署失敗或功能異常 | 高 | 先在測試環境完整驗證，凍結版本與設定 |
| 主機資源不足 | 效能差、UAT 失敗 | 中 | 上線前做容量驗證，預留 CPU/RAM 20~30% |
| 權限模型設計不清 | 使用者看不到資料或權限過大 | 高 | 先定角色矩陣，SIT 做完整權限案例測試 |
| 備份不可還原 | 表面有備份，實際無法救援 | 高 | 每月還原演練，納入驗收條件 |
| 文件分類失控 | Outline 上線後難搜尋、難治理 | 中 | 先設空間規則、模板、命名規範 |
| 使用者 adoption 低 | 上線後不用或回到舊流程 | 中 | UAT 前安排種子用戶、教育訓練、首頁導覽 |
| 客製化過多 | 拖慢時程、增加維運風險 | 高 | MVP 嚴控只做必要客製，其他列 P1/P2 |
| 單點故障 | APP 或 DB 故障即中斷服務 | 中 | MVP 先接受單點，但補齊備份/回滾；P2 再做 HA |

## 8.2 重點風險應對策略

### 1. 封閉網路供應鏈風險
- 先建立「可離線安裝清單」與「映像倉入庫流程」
- 所有第三方元件必須先經掃描與簽核
- 不要等到部署當天才補套件

### 2. 帳號整合風險
- 若銀行 AD 申請流程長，MVP 不要把 SSO 當唯一依賴
- 準備 local auth 過渡方案，保證 MVP 能先落地

### 3. 維運與稽核風險
- 備份、權限、日誌不能留到最後一週才補
- 這三項必須跟應用部署同步做

### 4. 時程壓縮風險
- 若 6 週內需強行交付，優先保留：
  - Homepage 基本入口
  - Plane 基本任務管理
  - Outline 基本知識庫
  - 權限 / 備份 / 監控
- 延後：報表、通知、深度整合、美化

# 10. 建議結論

對 5 人銀行 IT 團隊而言，這個 MVP 的成功關鍵不是「把三套系統都裝起來」，而是：

1. **先把封閉網路可部署性解決**
2. **先把權限、備份、監控、稽核打底**
3. **只交付最核心流程，不過度客製**
4. **用 6 週完成一個能上線、能維運、能稽核的版本**

建議採用以下落地順序：
- **第 1 優先**：環境、網路、TLS、帳權、備份
- **第 2 優先**：Plane 核心任務管理
- **第 3 優先**：Outline 核心知識庫
- **第 4 優先**：Homepage 單一入口與導覽
- **第 5 優先**：報表、通知、深度整合

若要進一步落地執行，下一步可補一份：
1. **實際甘特圖（Excel / Mermaid）**
2. **RACI 責任矩陣**
3. **正式環境部署架構圖**
4. **UAT 驗收案例清單**
